# 커밋·푸시 규칙

## 메시지

- 한국어로 작성한다.
- 한 줄, 50자 이내로 간결하게 작성한다.
- 기존 커밋 스타일을 따른다: `entry 정의`, `WAL과 Syncer 추가`.
- 무엇이 변경되었는지 기술한다. 이유/배경은 비자명할 때만 포함한다.
- `Co-Authored-By`, "Generated with Claude Code" 등 AI 생성 관련 문구는 절대 포함하지 않는다.

## 실행 시점

- `git commit`, `git push`는 사용자가 그 행위를 **명시적으로** 요청했을 때만 수행한다.
- 한 번의 승인은 **그 한 번**만 유효하다. 후속 작업의 커밋·푸시는 별도 승인을 받는다.
- 작업이 일단락되면 "커밋·푸시할까요?" 형태로 명시적으로 묻고 답을 기다린다.
- "커밋할까요?" 질문 직후 응답이 yes(또는 명백히 동등한 긍정)가 아니면 **모두 not-yes로 처리**한다 — 다른 코드 지시, 점검·재확인 요청, 질문, 무관한 메시지 모두 포함. 그 응답을 우선 수행한 뒤 작업이 다시 일단락되면 **다시 묻는다**.
- 직전 cycle이 "작성 → 묻고 → 커밋"으로 반복됐다는 이유로 다음 cycle의 승인을 추정하지 않는다. 매 cycle마다 새로운 명시 승인이 필요하다.
- "다음 단계로 진행해줘", "이어서 해줘", "이대로 가자" 같은 진행 지시는 **코드 작성·테스트까지만** 진행하라는 뜻이고 커밋 승인이 아니다.

## 스테이징

- 파일을 스테이징할 때 `git add -A` / `git add .` 대신 파일명을 개별 지정한다.
