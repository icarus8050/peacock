# 작업 진행 규칙

각 작업은 (1) 이해 → (2) 계획·승인 → (3) 구현 → (4) 리뷰·리팩토링·회고 순으로 진행한다. 사이클을 돌 때마다 규칙·스킬이 점진적으로 보강되도록 마지막 단계(회고)를 항상 수행한다.

## 1. 이해 — 추론 금지, 모호하면 질문

지시에 모호하거나 혼란스러운 부분이 있으면 **추측하여 작업하지 않는다**. 짧게라도 질문해 명확히 한 뒤 진행한다.

- 도메인 의도·디자인 선택이 갈리는 지점만 질문 대상.
- 단순 사실 확인(파일 경로, 함수 위치 등)은 코드를 직접 읽어 답한다 — 추론이 아니라 조사.
- 자주 빠지는 함정: 두 해석이 가능한 지시를 한쪽으로 단정, "사용자가 X를 의도했을 것"이라는 기대 매핑.

## 2. 계획·승인 — 구현 전에 합의

비-trivial 작업(여러 파일, 새 패키지, 공개 API 변경, 디자인 결정 포함)은 구현 전에 **계획을 세우고 사용자 승인을 받는다**.

계획에 포함할 것:
- 변경 대상 파일·심볼.
- 결정 사항이 갈리는 지점 — 옵션, 각 옵션의 장단, 추천과 근거.
- 단일 옵션이 자명하면 그렇게 적고 진행.

옵션 비교는 짧고 비교 가능한 형태로 제시한다 (표 또는 옵션 1/2 구조).

다중 커밋 작업이면 `.claude/plans/<topic>.md`에 결정사항·재개 절차를 기록한다.

자명한 단일 파일 수정(타입 1줄, 오타 등)은 계획 단계 생략 가능 — 단, 회피하려고 "트리비얼"로 분류하지 않는다.

## 3. 구현

합의된 계획대로 수행한다. 계획에서 벗어나는 결정이 필요하면 진행을 멈추고 다시 합의한다. 아키텍처 판단이 비자명하면 `Plan` sub-agent 호출.

## 4. 리뷰·리팩토링·회고 — 작업 종료 시 자동

작업이 일단락되면 사용자 별도 지시 없이도 다음 3단계를 자동으로 수행한다.

### 4.1 리팩토링 (`/refactor-scan`)

변경된 패키지(또는 새 소비자 포함)를 점검 — 추상화 수준 일치, 책임 분리, 정책 비대칭, 중복 등. 채택할 항목을 그 단계에서 정리.

### 4.2 코드 리뷰 (`code-reviewer` sub-agent)

비-trivial 변경(새 패키지, 공개 API 변경, ~50라인 이상)에서 `Agent` 툴로 호출 (`.claude/agents/code-reviewer.md`). fresh context에서 `refactor-scan`/`self-review` 체크리스트로 독립 검토.

리뷰 결과는 우선순위(High/Med/Low)로 분류하고 채택 항목을 적용. 보류 항목은 plan 또는 후속 커밋 메모에 남긴다.

### 4.3 회고 (`/retrospect`)

재사용 가능한 패턴이 발견되면 기존 규칙·스킬을 우선 보강한다. 새 스킬은 꼭 필요할 때만. 한 세션에 과도하게 부풀리지 않는다 (1~2개 이내). MEMORY.md의 stale 항목도 정리.

### 자가 학습 — 사용자가 지적하지 않아도

리뷰·리팩토링 중 **재사용 가능한 패턴**(같은 종류의 지적이 두 번 이상 나오거나, 도메인 무관한 일반 원칙으로 일반화 가능한 것)이 발견되면 사용자 요청 없이도:
- 기존 규칙 또는 스킬에 한 줄 추가/보강.
- 너무 좁은 1회성 패턴은 보강하지 않음 (over-fitting 회피).
- 변경분에 묶어 같은 commit으로 처리.

목적은 "다음 세션에서 같은 함정을 반복하지 않게" 만드는 것 — 자가 학습 루프를 닫는다.

### 작업 규모와 단계 적용

작업이 작아 일부 단계의 결과가 trivial이면 **결과만 짧게 보고한다 — 건너뛰지는 않는다**. 한 단계라도 의미 있는 발견이 있으면 사용자에게 보고하고 채택 여부를 묻는다.
