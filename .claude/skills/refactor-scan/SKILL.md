# 리팩토링 스캔

`$ARGUMENTS`에 지정된 경로(패키지 또는 파일)가 있으면 그 범위를, 없으면 `git status`로 변경된 Go 파일 전체를 대상으로 삼는다.

## 스코프 설정

변경된 파일 자체만 보지 않는다. **새 소비자가 등장한 경우**(예: 새 패키지가 기존 패키지를 import하기 시작), 의존 대상 패키지의 공개 API를 소비자 관점에서 한 번 더 훑는다. 경계 재조정이 필요한 문제는 패키지 경계를 넘어야 보인다.

**의존 패키지의 비공개 헬퍼도 한 번은 훑는다.** 새 패키지가 기존 패키지를 처음 import하기 시작하는 transition은 의존 대상 코드의 가독성을 점검할 자연스러운 트리거다. 공개 API의 시그니처만 보지 말고, 그 패키지의 비공개 함수까지 한 번은 들여다본다 — 특히 들여쓰기 깊이 점검(아래)은 공개/비공개 무관하게 적용한다.

### 들여쓰기 깊이 grep

스코프 안의 모든 Go 파일에 대해 **들여쓰기 3단계 이상의 함수 본체**를 찾아 후보로 등록한다. 빠른 휴리스틱:

```bash
# 12-space (3단계) 들여쓰기로 시작하는 줄을 가진 파일/줄을 찾는다.
grep -rn '^            [^[:space:]]' --include='*.go' <scope>
```

매칭된 파일은 함수 단위로 열어 `for` → `if err != nil` → `if errors.Is(...)` 류의 중첩이 의미 결정을 들고 있는지 본다. 해당 시 `go-style.md`의 "들여쓰기 평탄화"(루프 본체 헬퍼화 / early-return 평탄화 / 도메인 특수 케이스 호출부 이동) 후보로 보고. 이 점검은 변경되지 않은 비공개 헬퍼도 새 import transition 시점이면 대상.

## 점검 항목

각 파일에 대해 아래 관점으로 검토한다.

1. **책임 분리** — 한 타입이 두 가지 이상의 독립적 관심사를 품고 있는가? (예: 기록 + 스케줄링)
2. **중복** — 동일한 기본값/로직이 두 곳 이상에 박혀 있는가?
3. **미사용 / 데드 코드** — 선언되었지만 사용되지 않는 필드, 옵션, 콜백.
4. **인터페이스 / 타입 배치** — 인터페이스나 팩토리 함수가 **실제 소비자**가 있는 패키지에 있는가?
   - 인터페이스/팩토리의 호출처를 `grep`으로 확인한다.
   - 호출처가 전부 특정 외부 패키지 하나에 몰려 있다면, 해당 타입을 그 패키지로 이전할지 검토한다.
   - 특히 현재 위치가 "제공자 패키지"(실제로 그 타입을 구현/만족하는 타입이 있는 패키지)라면 더욱 의심한다.
   - 규칙 문서의 예시("X는 Y에 있어야 함")를 증거로 쓰지 말고, **규칙의 의도**("소비자 측 정의")가 실제 구조에 반영되었는지로 판단한다.
5. **에러 처리 일관성** — 래핑 방식, 에러 비교 관용구(`errors.Is` 등)가 프로젝트 내에서 통일돼 있는가?
6. **Zero value 활용** — 불필요한 명시적 초기화가 있는가?
7. **추상화 수준 일치** — 한 함수가 고수준 흐름(예: 정규화 → 재생 → 오픈 → 시작)과 저수준 세부(옵션 필드 조작, 바이트 조립, 플래그 분기 등)를 한꺼번에 품고 있는가? 섞여 있으면 저수준 블록을 이름 있는 헬퍼(예: `walOptionsFrom`)로 추출해 호출부가 한 레벨로 읽히게 한다. **들여쓰기 3단계+** 함수가 보이면 `go-style.md`의 "들여쓰기 평탄화" 세 패턴을 후보로 등록.
8. **문서-코드 동기화** — CLAUDE.md, 주석, 타입 문서가 실제 구현과 어긋나지 않는가?
9. **정책 비대칭** — 같은 데이터 구조에 대한 둘 이상의 호출자(또는 read/write 경로)가 같은 invariant 위반을 다르게 처리하고 있지 않은가? 의도된 분기라면 doc로 명시, 아니면 정렬한다. 예: writer는 자가 치유 / reader는 ErrCorrupt 같은 비대칭이 의도 없이 굳어졌는지 점검.
10. **버퍼링 / 성능 명백한 결함** — 명백히 불필요한 syscall, 할당 루프 등 (단, 벤치마크 없이 미세 최적화 제안 지양).

## 출력 형식

- 우선순위 순(High → Low)으로 번호 매긴 목록.
- 각 항목: **제목 / 문제 / 근거가 되는 파일:라인 / 제안하는 변경 / 트레이드오프 (있다면)**.
- 마지막에 추천 작업 순서 요약.

## 금지 사항

- 사용자의 승인 없이 코드를 수정하지 않는다. 제안만 한다.
- 벤치마크 없이 "성능 개선"을 최우선 근거로 제시하지 않는다.
- `go-style.md`의 규칙에서 벗어나는 제안은 하지 않는다.
