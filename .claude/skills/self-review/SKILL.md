# 셀프 리뷰

`refactor-scan`이 "무엇을 바꿀지"를 제안한다면, `self-review`는 "이미 바꾼 것이 깨끗한지"를 점검한다.

대상 범위: `$ARGUMENTS`에 지정된 경로가 있으면 그 범위, 없으면 `git diff`(스테이징 + unstaged)로 얻은 변경분 전체.

## 점검 항목

1. **gofmt 정렬 그룹** — struct literal / 구조체 필드 선언에 line-comment를 삽입해 정렬 그룹이 끊어진 곳이 없는가? 필요하면 inline 주석으로 전환한다.
2. **rename / 이동 위생** — 리네임하거나 패키지를 옮긴 식별자가
   - import된 패키지명과 겹쳐 섀도잉을 유발하지 않는가 (같은 스코프의 로컬 변수도 동시 리네임했는가)
   - 이전 이름/위치를 참조하는 orphan이 남아 있지 않은가 (`grep`로 확인)
   - **규칙 문서(`.claude/rules/*.md`)의 예시 코드에 옛 이름/위치가 박혀 있지 않은가** (go-style, commit, testing 등)
3. **에러 래핑 일관성** — 새로 추가/수정된 에러 반환점이 패키지 프리픽스로 래핑됐는가 (`fmt.Errorf("pkg: ...: %w", err)`).
4. **sentinel 주석** — `0`, `nil`, `""` 같은 값이 "다른 곳에 기본값을 위임" 등 비자명한 의미를 가진다면 inline 주석으로 명시.
5. **zero value 적합성** — 명시적으로 zero value로 초기화한 필드가 있는가 (불필요한 명시적 초기화 제거).
6. **go-style.md / commit.md 규칙 재확인** — 주석은 "왜"에만, 데드 코드 없음, 작업 번호/호출자 주석 없음.
7. **테스트 setup 어서션** — 회귀를 잡기 위해 추가한 테스트라면, 회귀 시점의 setup 전제(예: "고아 파일이 디렉터리 스캔에 잡히는 위치에 있다", "magic 바이트가 의도한 값")가 외부에서 관찰 가능한 어서션으로 코드에 박혀 있는가? 전제가 본문에서 인라인 주석으로만 남아 있다면 한 줄 어서션을 추가해 의도가 자기-증명되도록.
8. **빌드 / 테스트 / vet** — 아래 3개가 모두 통과하는가
   - `gofmt -l .` (출력 없어야 함)
   - `go vet ./...`
   - `go test ./...`
9. **문서 동기화** — 리네임/시그니처 변경이 CLAUDE.md의 API 기술과 일치하는가.

## 작업 방식

- 점검 항목별로 발견한 문제를 나열하고, 사소한 편집 위생 이슈는 즉시 수정한다 (설계 변경이 아니므로 승인 불필요).
- 설계/인터페이스 수준의 변경이 필요해 보이면 수정하지 말고 별도로 보고한다 (그건 `refactor-scan`의 영역).
- 마지막에 "수정한 항목 / 남은 의문" 두 줄로 요약한다.

## 금지 사항

- 범위를 벗어난 파일에는 손대지 않는다 (diff 밖 파일은 제안만).
- 벤치마크 없는 성능 최적화 변경 금지.
- `go-style.md` 규칙과 상충하는 변경 금지.
