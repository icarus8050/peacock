# 회고 및 스킬 추출

이 스킬은 "다음 세션의 내가 더 잘하도록" 만들기 위해, 방금 한 작업에서 **비자명한 패턴**을 뽑아 스킬·규칙·메모리에 반영한다. 매 작업 후가 아니라 **학습 가치가 있는 마디**에서 호출한다.

## 대상 범위

- 최근 대화 내용.
- 현재 브랜치의 `git log`(이번 세션 커밋 위주).
- 변경된 파일 및 인접 스킬/규칙 파일.

## 점검 질문 (우선순위 순)

1. **놓쳤다가 나중에 잡힌 문제가 있었는가?**
   - 근본 원인을 분석한다 (스코프 협소화? 규칙 예시 앵커링? 데이터 흐름 미추적?).
   - 같은 유형을 다음에 잡을 체크 항목을 **기존 스킬에 한 줄 추가**한다.
2. **사용자가 명시적으로 교정·확증한 지점이 있는가?**
   - 비자명한 교정/확증이면 auto-memory에 `feedback` 타입으로 저장.
3. **반복될 것 같은 결정 패턴이 있는가?**
   - 3회 미만: 관찰만 기록. 3회 이상: 규칙/스킬로 승격.
4. **기존 스킬의 체크 항목이 이번에 작동했는가?**
   - 미발동했다면 description/본문이 약함 → 트리거 조건 보강.
5. **규칙 문서와 실제 행동이 어긋났는가?**
   - 규칙이 약하면 예시·근거 보강. 규칙이 outdated면 갱신.
6. **CLAUDE.md와 실제 구조가 어긋났는가?**
   - Architecture / Rules 섹션 동기화.

## 산출물 우선순위

1. **기존 스킬/규칙 업데이트** — 가장 먼저 시도. 한 줄 추가로 해결되면 거기서 멈춘다.
2. **새 스킬 생성** — 1로 안 풀리고, 호출 트리거가 명확하며, 독립 절차로 분리 가능한 경우에만.
3. **메모리 저장** — 스킬·규칙으로 만들기엔 개인/프로젝트 맥락이 강한 항목.
4. **CLAUDE.md 업데이트** — 아키텍처/API 변경이 문서와 어긋날 때.

## 작업 방식

- 각 제안에 **왜 필요한가** (이번 세션의 실패/성공 사례)를 근거로 붙인다.
- 사용자 승인 후 반영. 자동 적용 금지.
- 추출한 패턴이 너무 세세하면 버린다 (YAGNI).
- 마지막에 "반영한 항목 / 관찰만 기록한 항목" 요약.

## 금지 사항

- 이미 있는 스킬/규칙을 무시하고 중복 생성 금지. 먼저 `ls .claude/skills/ .claude/rules/`로 확인.
- "방어적 미래 대비" 명목의 추상적 체크 항목 추가 금지.
- 한 세션에 너무 많은 신규 스킬 생성 금지 (1~2개 이내).
